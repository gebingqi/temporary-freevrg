# 网络协议最小长度检查子模板

## 适用 pattern

```text
missing-minimum-length-check-on-network-protocol-packet
```

## 漏洞机制

网络协议解析器从 UDP/DHCP/RADIUS 等报文中读取长度、偏移或属性长度字段后，仅检查是否超过缓冲区上界，或完全缺少非负/最小长度检查。攻击者可构造零、负数或过小长度，使循环、指针推进或属性读取发生越界、下溢或无限回溯。

抽象路径：

```text
packet length/offset -> loop or pointer advance -> missing minimum/nonnegative guard
```

## Source 建模建议

优先识别以下来源：

- 报文字节或头部字段解码出的长度。
- TLV 属性长度。
- 压缩域名指针偏移或递归解码长度。
- UDP 协议响应包中的 length 字段。

## Sink 建模建议

优先识别以下使用点：

- 以长度/偏移控制循环边界。
- 以长度/偏移进行指针推进。
- 基于长度读取协议属性。
- 基于长度更新解析位置。

## Sanitizer 建模建议

使用点前同一函数内出现以下检查，可视为有效 guard：

- `len < MIN`
- `len < 0`
- `pointed_len < 0`
- `alen < 2`
- `len < MIN || len > buffer_len`
- `remaining >= header_len && remaining >= declared_len`

只检查 `len > buffer_len` 属于单侧上界检查，不能视为完整 sanitizer。

## 负例

- 长度字段来自已验证的固定头部，调用前已有统一协议解码器保证最小长度。
- 每次解析前都检查剩余缓冲区至少包含 TLV 头部和声明长度。
- 长度只用于日志输出或错误消息，不影响读取、循环或指针推进。

## QL 形态建议

第一版可以优先识别函数内“上界有、下界无”的模式：

```ql
from Function f, Expr len, Stmt useSite
where
  isPacketLengthLike(len, f) and
  isParserUse(len, useSite, f) and
  not hasMinimumOrNonnegativeGuard(len, useSite, f)
select useSite, "Packet length or offset is used without a minimum-length or nonnegative check."
```

## few-shot anchors

| CVE | source | sink | sanitizer |
|-----|--------|------|-----------|
| CVE-2020-7461 | DHCP Option 119 search list | `pointed_len` controls domain tracing loop | `if (pointed_len < 0)` |
| CVE-2021-29629 | RADIUS response `Length` field | `len` controls attribute parsing | `if (len < POS_ATTRS || len > h->in_len)` |

## 验证重点

- 如果 vulnerable 版本 miss，优先检查 source 是否能识别协议解码变量而不是只识别 API 调用。
- 如果 fixed 版本仍报，优先补充双侧检查和非负检查。
- 如果误报多，要求 sink 必须影响解析位置、循环或缓冲区读取。
