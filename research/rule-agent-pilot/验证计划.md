# RuleAgent 试点验证计划

## 验证目标

验证目标不是证明规则已经可进入规则库，而是证明 RuleAgent 生成链路具备最小可行性：

```text
pattern -> rule_input.json -> candidate .ql -> compile -> vulnerable recall -> fixed false-positive check
```

## 不在本轮验证范围内

- 不做 test set 评估。
- 不做 FreeBSD-CURRENT 大规模扫描。
- 不处理多 CVE 综合 diff。
- 不引入随机切分或混合验证。
- 不实现官方新 SA 自动更新机制。

## 通用验证步骤

1. 读取试点目录下的 `rule_input.json`。
2. 由 RuleAgent 生成候选 `.ql`。
3. 使用 CodeQL CLI 执行编译检查。
4. 为每个成员 CVE 准备 vulnerable 版本 CodeQL database。
5. 在 vulnerable database 上运行候选 `.ql`，检查是否命中预期文件、函数或等价危险使用点。
6. 为每个成员 CVE 准备 fixed 版本 CodeQL database。
7. 在 fixed database 上运行候选 `.ql`，检查是否仍命中同一漏洞位置。
8. 记录结果和失败归因。

## 结果记录格式

建议每条规则生成后记录：

```json
{
  "pattern_id": "...",
  "query_path": "...",
  "compile": {
    "status": "pass|fail",
    "log": "..."
  },
  "recall": [
    {
      "cve": "...",
      "database": "...",
      "expected_file": "...",
      "expected_function": "...",
      "status": "hit|miss|not_run",
      "notes": "..."
    }
  ],
  "fixed_false_positive": [
    {
      "cve": "...",
      "database": "...",
      "status": "clean|still_reports|not_run",
      "notes": "..."
    }
  ],
  "decision": "repair|defer|candidate_pass"
}
```

## 试点 A：数组/描述符上界检查

Pattern:

```text
missing-array-bounds-check-on-network-controlled-index
```

历史召回目标：

| CVE | 文件 | 预期函数/位置 | 关键 sink | fixed guard |
|-----|------|---------------|-----------|-------------|
| CVE-2018-17161 | `libexec/bootpd/bootpd.c` | `handle_request` | `bp_htype` / `hwinfocnt` | `if (bp->bp_htype >= hwinfocnt)` |
| CVE-2019-5604 | `usr.sbin/bhyve/pci_xhci.c` | `pci_xhci_device_doorbell` | `dev->eps[epid]` | `if (epid == 0 || epid >= XHCI_MAX_ENDPOINTS)` |
| CVE-2021-29631 | `usr.sbin/bhyve/pci_virtio_9p.c` | `pci_vt9p_notify` | `vq_getchain` / `VT9P_MAX_IOV` | `assert(n >= 1 && n <= VT9P_MAX_IOV)` |

预期结果：

- vulnerable 版本中识别外部整数流入数组索引或描述符数组数量。
- fixed 版本中因存在上界/区间 guard 而不再报告同一位置。

## 试点 B：网络协议最小长度检查

Pattern:

```text
missing-minimum-length-check-on-network-protocol-packet
```

历史召回目标：

| CVE | 文件 | 预期函数/位置 | 关键 sink | fixed guard |
|-----|------|---------------|-----------|-------------|
| CVE-2020-7461 | `sbin/dhclient/options.c` | `find_search_domain_name_len` | `pointed_len` | `if (pointed_len < 0)` |
| CVE-2021-29629 | `lib/libradius/radlib.c` | `is_valid_response` | `len` / `POS_ATTRS` | `if (len < POS_ATTRS || len > h->in_len)` |

预期结果：

- vulnerable 版本中识别来自网络报文的长度/偏移只做上界或缺少非负/最小长度检查。
- fixed 版本中因存在非负或双侧长度检查而不再报告同一位置。

## 失败处理原则

- 编译失败：优先修 import、CodeQL API、语法和 predicate 类型。
- vulnerable miss：检查 source/sink 建模是否过窄，必要时回修子模板。
- fixed still reports：检查 sanitizer 建模是否过窄。
- 大量无关命中：检查 source 约束是否过宽，优先限制为网络包字段、MMIO/virtqueue 字段或协议长度字段。
- 若同一失败重复 3 次，暂停自动修复，转人工审查。
