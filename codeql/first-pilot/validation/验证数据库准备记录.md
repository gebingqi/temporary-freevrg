# 验证数据库准备记录

## 本轮结论

已为 5 个试点 CVE 准备 vulnerable/fixed 最小 CodeQL C/C++ database，共 10 个数据库。数据库位于：

```text
D:\桌面\学习\实习\first-ql-validation-package\minimal-validation-databases\db
```

这些数据库是机制验证库，不等同于完整 FreeBSD 历史源码版本数据库。2026-07-11 已完成 QL 归因、数据库重建和最终强制复测：5 个 vulnerable 库均命中预期危险使用点，5 个 fixed 库均为 0，首批两条规则通过机制级闭环验证。

## 路径与建库方式

- 正式交付目录已改为英文：`first-ql-validation-package`。
- 最小验证库目录已改为英文：`minimal-validation-databases`。
- 本机 `clang` 可用路径：`C:\Program Files\LLVM\bin\clang.exe`，版本 `22.1.8`。
- 10 份 harness 已用 clang 做过 `-std=c11 -c` 编译检查，均通过。
- CodeQL C/C++ `database create` 在中文 source/db 路径下会出现 extractor/finalize 失败；`subst Q:` 映射也不稳定。
- 实际建库使用真实 ASCII staging 路径：

```text
D:\freevrg-codeql-work\first-ql-validation-package\minimal-validation-databases
```

- 建库完成后，已将 finalize 成功的 database 和日志复制回项目交付目录。
- 建库命令采用 `--build-mode=none`，因为本机 CodeQL C/C++ extractor 配置支持该模式，且最小 harness 不需要真实 FreeBSD 构建系统。
- CVE-2021-29631 vulnerable/fixed harness 使用真实 `<assert.h>`，用于验证 CodeQL `Assertion.getAsserted()` 对 libc assert 宏的建模。
- PowerShell `-Force` 和 Linux smoke 脚本均使用 CodeQL `--rerun`，避免复用数据库内旧 BQRS。

## 失败归因与修正

| 现象 | 根因 | 修正 |
|------|------|------|
| 数组规则 fixed still_reports | `Expr.toString()` 在复合条件中退化为 `...`，无法按变量身份关联 guard | 改用 `VariableAccess.getTarget()`、关系运算 AST 和提前返回/有效分支结构 |
| CVE-2021-29631 旧命中无效 | `iov`、`&idx` 被当 source，`vq_getchain` source 调用被当 sink | 只跟踪整型返回变量 `n`，sink 改为其控制的描述符循环或数量敏感使用 |
| libc assert 未被验证 | 原 harness 把 `assert` 声明成普通函数 | 改用 `<assert.h>`，QL 同时支持 `Assertion.getAsserted()` 和函数式 fallback |
| CVE-2020-7461 vulnerable miss | 只匹配 `AssignExpr`，遗漏 `+=` | 使用 `Assignment` 基类并识别 `domain_name_len += pointed_len` |
| CVE-2021-29629 vulnerable miss | 循环、调用参数和 guard 依赖字符串匹配；`POS_ATTRS` 被混作 source | 以变量 AST 关联，协议常量仅作为阈值/位置锚点 |
| RADIUS fixed helper 误报 | 普通 `MD5Update` 参数 `len` 被无条件当作协议 source | 参数/字段 source 限制在协议解析语境，保留实际报文字节解码变量 |
| 强制复测仍返回旧结果 | `-Force` 只覆盖 SARIF，CodeQL 复用了旧 BQRS | analyze 命令加入 `--rerun` |

## 闭环判断

两条规则均通过 CodeQL 2.25.6 编译、5 个 vulnerable 机制召回和 5 个 fixed 清零，并已将同一修正同步到 RuleAgent 子模板和提示词草案。因此，它们足以支撑第一批“机制级 RuleAgent 闭环”。该结论不代表完整 FreeBSD 历史源码上的召回率或误报率；扩大样本或进入 PASS_WEAK pattern 前仍需真实历史版本数据库验证。
