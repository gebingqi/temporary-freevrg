# 首批 QL 验证矩阵

## 验证口径

每条规则必须先通过 CodeQL 编译，再进入 vulnerable/fixed database 验证。

验收口径：

- vulnerable database：应命中预期函数或等价危险使用点。
- fixed database：不应继续命中同一漏洞位置。
- 若 miss 或 still_reports，必须记录失败归因。

最终复测使用 CodeQL 2.25.6 和 `--rerun` 强制重新求值。五条 vulnerable SARIF 的实际位置依次为：`bp_htype` 数组访问第 28 行、`epid` 数组访问第 32 行、`n` 描述符循环第 74 行、`pointed_len` 长度累加第 39 行、`len - POS_ATTRS` 解析调用第 71 行。未出现 `slot - 1`、`iov`、`&idx`、协议常量或 helper 参数替代命中。

## 闭环结论

两条规则均满足“编译通过、vulnerable 正确 sink 命中、fixed 对应位置清零、建模约束回写子模板和 prompt 草案”。据此判定第一批机制级 RuleAgent 闭环成立。该结论仅覆盖 10 个最小验证数据库，不外推为完整 FreeBSD 历史源码召回率或误报率。
